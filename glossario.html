Atualização diária Últimos 5 meses Público · Privado · Grupo · INSS

Acompanha a evolução mensal e diária de abertura de contas PicPay por colaboradores elegíveis ao Benefits. Permite identificar crescimento por secretaria, setor e região do Brasil, com filtros interativos de mês, setor e estado.

Seções
📊
KPIs Mensais
Total de aberturas por mês nos últimos 5 meses, com variação m/m.
🏢
Top Secretarias
Ranking por volume de abertura, com filtro por setor e mês.
🗺️
Distribuição por Região
Aberturas agrupadas por região do Brasil.
📅
Evolução Diária
Curva dia-a-dia do mês selecionado.
⚡
Status de Antecipação
Quem antecipou, quem antecipou no dia de abertura e quem nunca antecipou.
Regras de negócio
Abertura de conta: Identificada por consumers.consumers.sent_bacen_at — data de envio ao BACEN. Registros com sent_bacen_at IS NULL são excluídos.
Deduplicação: Cada colaborador é contado uma vez via QUALIFY row_number() OVER (PARTITION BY collaborator_document ORDER BY collaborator_id DESC) = 1 — preserva o vínculo mais recente.
Segmentação de setor: account_id = 1420 → INSS | account_id IN (6,52,57,58,60,62,65,73,100,3170,3675) → Grupo | IS NULL → INSS | demais → flag_company_sector.
Janela temporal: Últimos 5 meses a partir do mês corrente (sent_bacen_at > current_date - 150 dias).
Comparativo MTD: Mês atual (dias 1–N) vs. mês anterior (dias 1–N), onde N é o dia de hoje. Garante comparação justa entre períodos iguais.
Status de antecipação: "Antecipou no dia" = sent_bacen_at = pri_antecipa | "Antecipou" = sent_bacen_at < pri_antecipa | "Nunca antecipou" = sem registro em anticipation_request.
Tabelas: benefits.sec_collaborators · consumers.sec_consumers · consumers.consumers · benefits.companies · benefits.accounts · benefits.anticipation_request
📄 abertura_de_contas.sql 
WITH base_docs AS (
  SELECT collaborator_document, company_id, collaborator_id
  FROM   benefits.sec_collaborators
  QUALIFY row_number() OVER (
    PARTITION BY collaborator_document
    ORDER BY collaborator_id DESC
  ) = 1
),
contas AS (
  SELECT consumer_id, cpf
  FROM   consumers.sec_consumers
  WHERE  cpf IN (SELECT collaborator_document FROM base_docs)
),
abertura AS (
  SELECT consumer_id,
         date_format(sent_bacen_at, 'yyyy-MM-dd') AS sent_bacen_at
  FROM   consumers.consumers
  WHERE  consumer_id IN (SELECT consumer_id FROM contas)
    AND  sent_bacen_at IS NOT NULL
  QUALIFY row_number() OVER (
    PARTITION BY consumer_id
    ORDER BY sent_bacen_at DESC
  ) = 1
),
pri_antecipa AS (
  SELECT consumer_id, min(date_format(created_at, 'yyyy-MM-dd')) AS pri_antecipa
  FROM   benefits.anticipation_request
  WHERE  request_status = 'FINISH'
    AND  consumer_id IN (SELECT DISTINCT consumer_id FROM abertura)
  GROUP BY 1
),
fim AS (
  SELECT
    a.collaborator_document,
    upper(f.account_name) AS account_name,
    f.account_id,
    -- Segmentação de setor
    CASE
      WHEN f.account_id = 1420                                           THEN 'INSS'
      WHEN f.account_id IN (6,52,57,58,60,62,65,73,100,3170,3675) THEN 'grupo'
      WHEN f.account_id IS NULL                                          THEN 'INSS'
      ELSE  coalesce(e.flag_company_sector, 'private')
    END AS setor,
    -- Região geográfica
    CASE
      WHEN e.address_state IN ('AC','AP','AM','PA','RO','RR','TO')              THEN 'Norte'
      WHEN e.address_state IN ('AL','BA','CE','MA','PB','PE','PI','RN','SE') THEN 'Nordeste'
      WHEN e.address_state IN ('DF','GO','MT','MS')                        THEN 'Centro-Oeste'
      WHEN e.address_state IN ('ES','MG','RJ','SP')                        THEN 'Sudeste'
      WHEN e.address_state IN ('PR','RS','SC')                            THEN 'Sul'
      ELSE 'Outra/Não Identificada'
    END AS regiao_brasil,
    c.sent_bacen_at,
    -- Status de antecipação
    CASE
      WHEN d.consumer_id IS NOT NULL AND c.sent_bacen_at = d.pri_antecipa THEN 'Antecipou dia abertura'
      WHEN d.consumer_id IS NOT NULL AND c.sent_bacen_at < d.pri_antecipa  THEN 'Antecipou'
      ELSE 'Nunca antecipou'
    END AS fl_antecipacao
  FROM       base_docs       a
  LEFT JOIN  contas          b  ON a.collaborator_document = b.cpf
  LEFT JOIN  abertura        c  ON b.consumer_id           = c.consumer_id
  LEFT JOIN  pri_antecipa    d  ON b.consumer_id           = d.consumer_id
  LEFT JOIN  benefits.companies e ON a.company_id          = e.company_id
  LEFT JOIN  benefits.accounts  f ON e.account_id          = f.account_id
)
SELECT
  date_format(sent_bacen_at, 'yyyy-MM')    AS mes,
  date_format(sent_bacen_at, 'yyyy-MM-dd') AS dia,
  setor,
  Secretaria                               AS secretaria,
  regiao_brasil                            AS regiao,
  count(*)                                 AS count
FROM fim
WHERE date_format(sent_bacen_at, 'yyyyMM') > date_format(current_date() - 150, 'yyyyMM')
  AND sent_bacen_at IS NOT NULL
GROUP BY 1, 2, 3, 4, 5
ORDER BY 1, 2, 3

MTD Atualização diária 6 queries paralelas 2 visões

Painel operacional com métricas MTD de antecipações de salário. Duas visões: todas as antecipações e excluindo GDF e RJ. Cada visão apresenta 6 indicadores por setor com comparativo ao mesmo período do mês anterior.

Indicadores (por setor: Total · Público · Privado · Grupo · INSS)
👥
Vidas
Colaboradores únicos que anteciparam no mês (DISTINCT consumer_id).
📦
Antecipações
Total de transações concluídas com request_status = 'FINISH'.
💰
TPV (R$)
Volume financeiro total (SUM(request_value)).
🎫
Ticket Médio (R$)
TPV ÷ antecipações — calculado no frontend.
🏦
Abertura de Contas
Contas abertas no mesmo período MTD.
🌱
Vidas Novas
1ª antecipação ever do colaborador dentro do mês atual.
Regras de negócio
MTD comparativo: Dias 1–N do mês atual vs. dias 1–N do mês anterior, onde N = dia de hoje. Filtro: day(created_at) <= N.
Status de conclusão: Somente request_status = 'FINISH' é contabilizado.
Vidas novas: Colaborador com primeira antecipação ever no mês vigente — exclui quem já tinha created_at < inicio_do_mes via subquery.
Visão sem GDF/RJ: Filtra account_id NOT IN (892, 1817). GDF (892) e Estado do RJ (1817) têm margem zerada por bloqueio sistêmico e distorcem médias nacionais.
Execução: 6 queries em paralelo (antecip · novas · abertura × mês atual e anterior). Autenticação via OAuth CLI, perfil "picpay".
Tabelas: benefits.anticipation_request · benefits.companies · benefits.accounts · benefits.sec_collaborators · consumers.sec_consumers · consumers.consumers
📄 antecipacoes.sql — 3 queries rodadas em paralelo (mês atual e anterior) 
-- Query 1: Volume de antecipações por setor
SELECT
  CASE
    WHEN f.account_id = 1420                                           THEN 'INSS'
    WHEN f.account_id IN (6,52,57,58,60,62,65,73,100,3170,3675) THEN 'grupo'
    WHEN f.account_id IS NULL                                          THEN 'INSS'
    ELSE  coalesce(e.flag_company_sector, 'private')
  END AS setor,
  count(distinct ar.consumer_id)                  AS vidas,
  count(*)                                         AS antecipacoes,
  round(sum(cast(ar.request_value AS double)), 2) AS tpv
FROM       benefits.anticipation_request ar
LEFT JOIN  benefits.companies  e ON ar.company_id  = e.company_id
LEFT JOIN  benefits.accounts   f ON e.account_id   = f.account_id
WHERE  ar.request_status = 'FINISH'
  AND  ar.created_at >= '<ini>'   -- 1º dia do mês
  AND  ar.created_at <  '<fim>'   -- 1º dia do mês seguinte
  AND  day(ar.created_at) <= <N>  -- dia de hoje (MTD)
GROUP BY 1

-- Query 2: Vidas novas (1ª antecipação ever no mês)
SELECT
  /* mesmo CASE setor acima */ setor,
  count(distinct ar.consumer_id) AS vidas_novas
FROM       benefits.anticipation_request ar
LEFT JOIN  benefits.companies e ON ar.company_id = e.company_id
LEFT JOIN  benefits.accounts  f ON e.account_id  = f.account_id
WHERE  ar.request_status = 'FINISH'
  AND  ar.created_at >= '<ini>'
  AND  ar.created_at <  '<fim>'
  AND  day(ar.created_at) <= <N>
  AND  ar.consumer_id NOT IN (
    SELECT DISTINCT consumer_id
    FROM   benefits.anticipation_request
    WHERE  request_status = 'FINISH'
      AND  created_at < '<ini>'  -- nunca antecipou antes do mês
  )
GROUP BY 1

-- Query 3: Abertura de contas no período
WITH base AS (
  SELECT sc.collaborator_document, sc.company_id
  FROM       benefits.sec_collaborators sc
  INNER JOIN consumers.sec_consumers cc ON cc.cpf         = sc.collaborator_document
  INNER JOIN consumers.consumers     c  ON c.consumer_id  = cc.consumer_id
  WHERE  c.sent_bacen_at >= '<ini>'
    AND  c.sent_bacen_at <  '<fim>'
    AND  day(c.sent_bacen_at) <= <N>
    AND  c.sent_bacen_at IS NOT NULL
  QUALIFY row_number() OVER (
    PARTITION BY sc.collaborator_document
    ORDER BY sc.collaborator_id DESC
  ) = 1
)
SELECT
  /* mesmo CASE setor */ setor,
  count(*) AS abertura_contas
FROM       base b
LEFT JOIN  benefits.companies e ON b.company_id = e.company_id
LEFT JOIN  benefits.accounts  f ON e.account_id = f.account_id
GROUP BY 1

Canal PIX Atualização diária 2 queries paralelas

Recorte exclusivo das antecipações cujo destino é PIX. Mesma estrutura do relatório de antecipações (6 indicadores × 4 setores + MTD), porém filtrado por cashin_destiny = 'PIX'. Após gerado, regera automaticamente o Dashboard TV.

Regras de negócio
Filtro principal: cashin_destiny = 'PIX' aplicado em benefits.anticipation_request. Demais destinos (TED, conta PicPay etc.) são excluídos.
Vidas novas PIX: CTE pri_antecipacao identifica o MIN(created_at) por consumer_id em todo o histórico PIX. Flag fl_pri = 1 = nunca antecipou via PIX antes.
Query unificada: 1 única query retorna os dois períodos (mês atual e anterior) via flags fl_cur e fl_prv no SELECT — reduz de 6 para 2 queries no total.
Sincronização TV: Após gerar pix.html, o script lê antecipacoes.html do disco e regera tv.html combinando os dois datasets. Os três arquivos sobem no mesmo commit.
Tabelas: benefits.anticipation_request (filtro: cashin_destiny = 'PIX') · benefits.companies · benefits.accounts · consumers.sec_consumers · consumers.consumers
📄 pix.sql — query unificada (cur + prv em uma passagem) 
WITH pri_antecipacao AS (
  -- Identifica a 1ª antecipação PIX ever por consumer
  SELECT consumer_id, min(created_at) AS dt_pri
  FROM   benefits.anticipation_request
  WHERE  request_status = 'FINISH'
  GROUP BY 1
),
fim AS (
  SELECT
    /* CASE setor (mesma lógica dos demais relatórios) */
    CASE
      WHEN f.account_id = 1420                                           THEN 'INSS'
      WHEN f.account_id IN (6,52,57,58,60,62,65,73,100,3170,3675) THEN 'grupo'
      WHEN f.account_id IS NULL                                          THEN 'INSS'
      ELSE  coalesce(e.flag_company_sector, 'private')
    END AS setor,
    ar.consumer_id,
    ar.request_value,
    -- Flag: primeira antecipação ever desse consumer?
    CASE WHEN pa.consumer_id IS NULL THEN 0 ELSE 1 END AS fl_pri,
    -- Flag: pertence ao mês atual (MTD)?
    CASE WHEN date_format(ar.created_at, 'yyyy-MM') = '<cur_mes>'
              AND day(ar.created_at) <= <N> THEN 1 ELSE 0 END AS fl_cur,
    -- Flag: pertence ao mês anterior (mesmo período)?
    CASE WHEN date_format(ar.created_at, 'yyyy-MM') = '<prev_mes>'
              AND day(ar.created_at) <= <N> THEN 1 ELSE 0 END AS fl_prv
  FROM       benefits.anticipation_request ar
  LEFT JOIN  benefits.companies  e  ON ar.company_id  = e.company_id
  LEFT JOIN  benefits.accounts   f  ON e.account_id   = f.account_id
  LEFT JOIN  pri_antecipacao     pa ON ar.consumer_id = pa.consumer_id
                                    AND ar.created_at  = pa.dt_pri
  WHERE  ar.request_status = 'FINISH'
    AND  ar.cashin_destiny = 'PIX'       -- <-- filtro principal
    AND  ar.created_at >= '<prev_ini>'  -- inicia no mês anterior (cobre os dois)
)
SELECT
  setor,
  count(distinct CASE WHEN fl_cur=1 THEN consumer_id END)               AS vidas_cur,
  count(CASE WHEN fl_cur=1 THEN 1 END)                                  AS antecip_cur,
  round(sum(CASE WHEN fl_cur=1 THEN request_value ELSE 0 END), 2)       AS tpv_cur,
  count(distinct CASE WHEN fl_cur=1 AND fl_pri=1 THEN consumer_id END)  AS novas_cur,
  count(distinct CASE WHEN fl_prv=1 THEN consumer_id END)               AS vidas_prv,
  count(CASE WHEN fl_prv=1 THEN 1 END)                                  AS antecip_prv,
  round(sum(CASE WHEN fl_prv=1 THEN request_value ELSE 0 END), 2)       AS tpv_prv,
  count(distinct CASE WHEN fl_prv=1 AND fl_pri=1 THEN consumer_id END)  AS novas_prv
FROM fim
GROUP BY 1
ORDER BY 1

Análise Profunda RFM Funil de Conversão Atualização automática disponível

Dashboard analítico de propensão à antecipação. Vai além dos volumes operacionais: mapeia quem antecipa, quando antecipa, para onde vai o dinheiro e qual o perfil de quem ainda não antecipou. Todos os indicadores têm filtro por setor (Total · Público · Grupo · Privado). Possui botão de atualização de dados integrado ao servidor local.

Seções
🔢
KPIs Hero
8 cards (Total Vidas, Com Conta, Com Margem, MAU 30d, Conta Sem Margem, Antecipando, Sem Conta, MAU Sem Margem) com filtro interativo por setor (Total · Público · Grupo · Privado).
🔽
Funil Real de Conversão
1,75M elegíveis → conta → margem → MAU → antecipando. % de conversão em cada etapa.
🎪
Funil de Campanha MKT
Análise dinâmica de funil por activity_name (envio único) ou journey_name (jornada completa). Retorna Enviados → Entregues → Abertos → Clicados → Anteciparam. Suporta múltiplas campanhas com comparativo automático e filtro por setor.
🏢
Análise por Setor
Mini-funil Público · Grupo · Privado com conversão conta → margem → antecipação. Os KPIs do topo também têm filtro por setor — clique nas abas Total · Público · Grupo · Privado para ver os indicadores do setor selecionado.
📅
Evolução Mensal
Tendência histórica por mês, com abas por setor.
🌡️
Sazonalidade
Heatmap por dia do mês — pico no dia 10. Filtro dinâmico por setor.
🧮
Perfil RFM
Recência, Frequência e Valor por setor. Segmenta antecipadores frequentes vs. pontuais.
📍
Destino do Pagamento
Para onde vai o dinheiro: PIX, TED, conta PicPay etc. Evolução temporal por setor.
Regras de negócio
Funil correto: Sem filtro has_anticipation_enabled e sem resignation_date IS NULL. Inclui colaboradores desligados — alinhado com o número real do Dashboard TV.
MAU: Calculado com D-7 (7 dias de estabilidade) usando is_mau_app_30 de consumers.daily_consumers_labels_and_metrics.
Funil MKT (dinâmico): Cloudflare Worker → GitHub Actions → Databricks. Dois modos: Activity (cole o activity_name para um envio específico) e Journey (cole o journey_name para ver funil consolidado de toda a jornada + botões por step). Cruza marketing.consumers_campaigns_communications com benefits.anticipation_request em janela de 24 horas após o envio. Cache local: 2ª busca é instantânea.
4 visões de antecipação no funil:  📊 Total (multi) — conta todas as antecipações nas 24h pós campanha (mesmo consumer pode contar N vezes) · 👤 Únicos — conta consumers distintos que anteciparam (1 pessoa = 1, mesmo que tenha antecipado várias vezes) · 🆕 1ª vez — consumers que anteciparam nas 24h pós campanha E a primeira antecipação da vida ocorreu após o envio (nunca tinham antecipado antes da campanha — novatos absolutos) · 🔄 Recompra — consumers que anteciparam nas 24h pós campanha E já tinham antecipado antes do envio (reativação / recompra). Novatos + Recompra = Únicos — as duas visões são mutuamente exclusivas, determinadas pela CTE sent_at_ref (menor sent_at por consumer como referência temporal).
Campo correto de valor: anticipation_request.request_value (não anticipation_value — inexistente nessa tabela). Campo de margem: sec_wage_advance_collaborator_margins.anticipation_value.
Botão "🔄 Atualizar dados": Presente no header do relatório. Dispara POST localhost:5001/atualizar/score, executa queries Databricks ao vivo e recarrega a página automaticamente ao concluir. Requer servidor local na porta 5001.
Tabelas: benefits.anticipation_request · benefits.collaborators · benefits.companies · benefits.accounts · benefits.sec_wage_advance_collaborator_margins · consumers.consumers · consumers.daily_consumers_labels_and_metrics · marketing.consumers_campaigns_communications
📄 score_funil.sql — funil principal de conversão 
-- Funil real de conversão (base do Score)
WITH
-- Total de vidas elegíveis no Benefits
elegíveis AS (
  SELECT count(distinct sc.collaborator_document) AS total
  FROM   benefits.sec_collaborators sc
),
-- Colaboradores com conta PicPay aberta
com_conta AS (
  SELECT count(distinct cc.consumer_id) AS total
  FROM       benefits.sec_collaborators sc
  INNER JOIN consumers.sec_consumers   cc ON cc.cpf        = sc.collaborator_document
  INNER JOIN consumers.consumers       c  ON c.consumer_id = cc.consumer_id
  WHERE  c.sent_bacen_at IS NOT NULL
),
-- Com margem ativa de antecipação
com_margem AS (
  SELECT count(distinct m.consumer_id) AS total
  FROM   benefits.sec_wage_advance_collaborator_margins m
  WHERE  m.is_enabled = true
    AND  m.anticipation_value > 0
),
-- MAU com margem (D-7 para estabilidade)
mau_margem AS (
  SELECT count(distinct lm.consumer_id) AS total
  FROM       consumers.daily_consumers_labels_and_metrics lm
  INNER JOIN benefits.sec_wage_advance_collaborator_margins m
          ON m.consumer_id = lm.consumer_id
         AND m.is_enabled = true
  WHERE  lm.metric_date = date_sub(current_date(), 7)
    AND  lm.is_mau_app_30 = true
),
-- Antecipando no mês corrente
antecipando AS (
  SELECT count(distinct ar.consumer_id) AS total
  FROM   benefits.anticipation_request ar
  WHERE  ar.request_status = 'FINISH'
    AND  date_format(ar.created_at, 'yyyy-MM') = date_format(current_date(), 'yyyy-MM')
  -- Sem filtro has_anticipation_enabled — conta todos os finalizados
)
SELECT
  e.total  AS elegiveis,
  c.total  AS com_conta,
  m.total  AS com_margem,
  mm.total AS mau_com_margem,
  a.total  AS antecipando
FROM elegíveis e, com_conta c, com_margem m, mau_margem mm, antecipando a

Infra · Dados Atualização diária às 8h 11 tabelas · 11 queries paralelas

Painel de saúde das fontes de dados. Monitora a data da última atualização das 11 tabelas Databricks usadas pelos relatórios Benefits e emite alertas de atraso ou problema.

🔒

Lógica especial — alerta por variação de COUNT de linhas.
🧑

⚡

📣

Regras de negócio
Critérios de status: ✅ Atualizado = ≤ 26h | ⚠️ Atraso = 26h–72h | ❌ Problema = > 72h ou erro.
benefits.accounts: Tabela sem movimentação diária. Alerta baseado em variação de COUNT(*) vs. snapshot salvo na execução anterior.
Parser de datas: Databricks retorna ISO com sufixo T, Z e milissegundos (ex: 2026-04-02T07:51:19.114Z). O script remove esses sufixos antes de calcular o delta em horas.
📄 monitor_tabelas.sql — executada para cada tabela (11 queries em paralelo) 
-- Padrão executado para cada tabela monitorada
-- Substitua <tabela> e <coluna_data> conforme a tabela

SELECT
  max(<coluna_data>) AS max_date,
  count(*)           AS total_rows
FROM <tabela>

-- Exemplos reais:
-- benefits.collaborators       → max(updated_at)
-- consumers.consumers          → max(sent_bacen_at)   ← NÃO updated_at
-- benefits.anticipation_request→ max(created_at)
-- daily_consumers_labels_...   → max(metric_date)     ← NÃO ref_date
-- marketing.consumers_campaigns→ max(sent_at)
-- picpay.picpay.all_transactions→ max(created_at)     ← schema duplo

-- Lógica de status (Python, pós-query):
-- delta = (now - max_date).total_seconds() / 3600
-- ≤ 26h  → "Atualizado"
-- 26-72h → "Atraso"
-- > 72h  → "Problema"

-- Lógica especial para benefits.accounts:
-- Não usa max(updated_at) — a tabela não muda diariamente
-- Compara COUNT(*) atual vs. snapshot JSON salvo na execução anterior
-- diff > 0  → "+N linhas adicionadas" (status OK)
-- diff == 0 → "Estável"             (status OK)
-- diff < 0  → "N linhas removidas"  (status ERRO)

PIX
Recorte exclusivo de antecipações com destino PIX.
Regras de negócio
Geração automática: Regerado por atualizar_antecipacoes.py (após gerar antecipacoes.html) e por atualizar_pix.py (após gerar pix.html). Nunca fica desatualizado.
Sincronização PIX: Ao atualizar o PIX, o script lê antecipacoes.html do disco (DATA) e combina com os dados PIX recém-calculados (DATA_PIX). Os três arquivos sobem no mesmo commit.
Auto-refresh: Recarrega a cada 5 minutos via setInterval, mas somente quando não está em fullscreen — evita saída involuntária do modo TV.
Timer de slides: setTimeout simples de 60s (não setInterval) — sem deriva acumulada de tempo entre slides.
Gerado a partir de: antecipacoes.html (DATA + DATA2) e pix.html (DATA_PIX) — nenhuma query adicional é executada.
